# 技术分析与选型决策：OSW Studio vs Assistant UI

## 1. 任务背景
用户要求对新演示项目进行技术选型，并在 `osw-studio` (当前项目，基于 Next.js Standalone + 原生 Fetch) 与 `ling-model-judge` (参考项目，基于 Assistant UI + LangGraph) 之间做出决策。

## 2. 原始指示 (User Instruction)
> 我面临一个技术选型：我们是像 osw-studio 一样从头构建页面？还是像那个项目一样使用 assistant-ui 构建项目？
> 这两者最大的区别在 chat 功能的实现上。assistant-ui 看起来好像提供了更简单的推理 api 集成，还能让我使用 langgraph。
> 我计划在我们新创建的项目中也使用 langchain 或者 langgraph 实现多模型工作流。
> 你帮我做个技术选型。过程中，你要着重分析 osw-studio 这个项目的推理 api 对接，以及推理结果如何管理为上下文。
> 你还要分析，osw-studio 项目是否方便对接 langchain 或者 langgraph。

## 3. 分析目标
1.  **OSW Studio 分析**：
    -   推理 API (`app/api/generate`) 的对接机制。
    -   推理结果上下文 (`LLMMessage[]`) 的管理方式。
    -   对接 LangChain/LangGraph 的可行性与成本。
2.  **Assistant UI (Ling Model Judge) 分析**：
    -   Chat 功能的实现复杂度。
    -   与 LangGraph 的集成优势。
    -   UI 定制的灵活性（针对我们要做的 Web Preview 和 Write Editor）。
3.  **最终决策**：基于新 Demo 的三大需求（Chat, Web, Write）和多模型工作流规划，给出推荐方案。

## 4. 研究计划
- [ ] 深入分析 `osw-studio` 的上下文管理逻辑。
- [ ] 探查 `ling-model-judge` 的 Assistant UI 集成方式。
- [ ] 对比两者的优劣势。
- [ ] 产出选型结论。
